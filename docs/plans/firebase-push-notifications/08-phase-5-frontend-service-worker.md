# Fase 5 — Frontend Service Worker para FCM

**Objetivo:** crear el `firebase-messaging-sw.js` que recibe pushes con la app cerrada y muestra notificaciones nativas del SO. Este es **el archivo crítico** que hace que la app cerrada reciba notificaciones.

**Esfuerzo estimado:** 0.5 día.

---

## 5.1 — Por qué un SW separado

Recap de la decisión cerrada (D7 en `00-overview.md`):

- `sw.js` (existente) → existe SOLO para PWA installability. Network-only pass-through. Scope `/`.
- `firebase-messaging-sw.js` (nuevo) → existe SOLO para recibir pushes FCM en background. Scope `/firebase-cloud-messaging-push-scope`.

Coexisten sin conflicto porque tienen scopes distintos. Cada uno es registrado por su dueño:
- `sw.js` → registrado en `main.ts` con `navigator.serviceWorker.register('/sw.js', { scope: '/' })`.
- `firebase-messaging-sw.js` → registrado automáticamente por el SDK de Firebase al llamar `getToken()`.

---

## 5.2 — Restricción crítica: el SW no puede leer `environment.ts`

El SW corre en un contexto sin acceso al bundle Angular. No puede importar TS, no tiene `process.env`, no tiene `import.meta`.

Tres opciones para inyectar la config Firebase:

| Opción | Pros | Contras |
|---|---|---|
| Hardcodear la config en el SW | Simple | Un archivo distinto por entorno. Si cambia el projectId, hay que tocar el SW. |
| Pasar la config vía query string al registrar | Funciona | Riesgo de URLs expuestas en el log; el SW puede caché incorrectamente. |
| **Generar el SW en build-time desde `environment.ts`** | Una sola fuente de verdad. Cero deriva. | Requiere un script Node que ejecutar antes de `ng build`. |

**Decisión cerrada (D-SW1):** opción 3 — generación en build-time.

---

## 5.3 — Script generador

### Archivo nuevo: `frontend/scripts/generate-firebase-sw.js`

```js
/**
 * Generates `public/firebase-messaging-sw.js` with the Firebase config
 * matching the active environment. Runs before `ng build` and `ng serve`.
 *
 * Reads from environment files based on NODE_ENV / configuration:
 *   - default: src/environments/environment.ts
 *   - production: src/environments/environment.prod.ts
 *
 * Why a generated file: the SW cannot import TypeScript or read env vars
 * at runtime. Build-time codegen keeps a single source of truth (env.ts)
 * and avoids drift between Angular and the SW.
 */

const fs = require('fs');
const path = require('path');

const isProd = process.env.NODE_ENV === 'production' || process.argv.includes('--prod');
const envFile = isProd ? 'environment.prod.ts' : 'environment.ts';
const envPath = path.join(__dirname, '..', 'src', 'environments', envFile);

if (!fs.existsSync(envPath)) {
  console.error(`[generate-firebase-sw] Cannot find ${envPath}`);
  process.exit(1);
}

const envSource = fs.readFileSync(envPath, 'utf8');

// Naive but sufficient: extract the firebase block. We expect the format
// `firebase: { ... }` with primitive values. If the format changes, tweak here.
const firebaseMatch = envSource.match(/firebase:\s*\{([^}]+)\}/);
if (!firebaseMatch) {
  console.error('[generate-firebase-sw] No "firebase: {...}" block found in env file');
  process.exit(1);
}

// Extract the keys we need (apiKey, projectId, messagingSenderId, appId, authDomain)
const extract = (key) => {
  const m = firebaseMatch[1].match(new RegExp(`${key}:\\s*['"]([^'"]+)['"]`));
  return m ? m[1] : '';
};

const config = {
  apiKey: extract('apiKey'),
  authDomain: extract('authDomain'),
  projectId: extract('projectId'),
  storageBucket: extract('storageBucket'),
  messagingSenderId: extract('messagingSenderId'),
  appId: extract('appId'),
};

if (!config.projectId || !config.messagingSenderId) {
  console.error('[generate-firebase-sw] projectId and messagingSenderId are mandatory');
  process.exit(1);
}

const swContent = `/**
 * AUTO-GENERATED by frontend/scripts/generate-firebase-sw.js
 * Do NOT edit by hand. To change config, edit environment.ts and rebuild.
 *
 * This Service Worker handles Firebase Cloud Messaging in the background:
 *  - Receives pushes when the app is closed or the tab is in background.
 *  - Displays the OS-native notification using the payload's
 *    \`notification\` field. The Firebase SDK handles this automatically.
 *  - On notification click, focuses an open tab or opens a new one,
 *    navigating to \`data.url\`.
 *
 * Coexistence note: this SW has a distinct scope from /sw.js (the PWA
 * installability SW), so they do not interfere.
 *
 * Generated for projectId: ${config.projectId}
 */

importScripts('https://www.gstatic.com/firebasejs/10.13.2/firebase-app-compat.js');
importScripts('https://www.gstatic.com/firebasejs/10.13.2/firebase-messaging-compat.js');

firebase.initializeApp(${JSON.stringify(config, null, 2)});

const messaging = firebase.messaging();

/**
 * Background message handler.
 *
 * When the payload includes a \`notification\` field, the Firebase SDK
 * shows the OS notification automatically — we do NOT need to call
 * showNotification() ourselves. Defining onBackgroundMessage explicitly
 * is required to keep custom logging / analytics paths if needed later.
 */
messaging.onBackgroundMessage((payload) => {
  // No-op: SDK shows the notification using payload.notification.
  // Logged for debugging:
  console.log('[firebase-messaging-sw] Background message received:', payload);
});

/**
 * Notification click — same UX as /sw.js: focus an existing tab if any,
 * else open a new one. Target URL travels in payload.data.url.
 *
 * Duplicated from /sw.js by design (decision D8). Both SWs share the
 * same UX so that whichever shows the notification handles the click
 * consistently.
 */
self.addEventListener('notificationclick', (event) => {
  event.notification.close();

  const targetUrl =
    (event.notification.data && (event.notification.data.url || event.notification.data.FCM_MSG?.data?.url)) ||
    '/';

  event.waitUntil((async () => {
    const allClients = await self.clients.matchAll({
      type: 'window',
      includeUncontrolled: true,
    });

    for (const client of allClients) {
      if ('focus' in client) {
        await client.focus();
        if ('navigate' in client) {
          try { await client.navigate(targetUrl); } catch { /* ignore */ }
        }
        return;
      }
    }

    if (self.clients.openWindow) {
      await self.clients.openWindow(targetUrl);
    }
  })());
});
`;

const outputPath = path.join(__dirname, '..', 'public', 'firebase-messaging-sw.js');
fs.writeFileSync(outputPath, swContent, 'utf8');
console.log(`[generate-firebase-sw] Generated ${outputPath}`);
console.log(`[generate-firebase-sw] Project: ${config.projectId}`);
`;
```

> Nota: el último triple-backtick es del bloque de markdown, no parte del JS. El JS termina en la línea con `console.log(...)` que cierra con backtick.

## 5.4 — Hookear el script en `package.json`

**Archivo:** `frontend/package.json`

Agregar al bloque `scripts`:

```json
{
  "scripts": {
    "ng": "ng",
    "prestart": "node scripts/generate-firebase-sw.js",
    "start": "ng serve",
    "prebuild": "node scripts/generate-firebase-sw.js",
    "build": "ng build",
    "prebuild:prod": "node scripts/generate-firebase-sw.js --prod",
    "build:prod": "ng build --configuration production",
    "watch": "ng build --watch --configuration development",
    "test": "ng test"
  }
}
```

Los hooks `prestart`, `prebuild` y `prebuild:prod` los ejecuta npm automáticamente antes del script principal.

## 5.5 — Asegurar que el archivo se sirve desde la raíz

El SW **DEBE** ser accesible en `/firebase-messaging-sw.js` (raíz absoluta del dominio). Firebase lo busca ahí por convención.

**Verificación en `angular.json`:**

```json
{
  "projects": {
    "tubus-express": {
      "architect": {
        "build": {
          "options": {
            "assets": [
              {
                "glob": "**/*",
                "input": "public"
              }
            ]
          }
        }
      }
    }
  }
}
```

Si `public/` está en la lista de assets (que ya debería estar para `sw.js`, `manifest.webmanifest`, etc.), el SW generado se servirá automáticamente desde la raíz.

**Verificación en runtime:** abrir `http://localhost:4200/firebase-messaging-sw.js` en el navegador → debe devolver el JS, NO el `index.html`.

## 5.6 — Ignorar el archivo generado en git

**Archivo:** `frontend/.gitignore`

Agregar:

```
# Auto-generated by scripts/generate-firebase-sw.js
public/firebase-messaging-sw.js
```

Razón: el archivo se regenera en cada build. Versionarlo causa conflictos cada vez que cambia la config Firebase.

## 5.7 — Verificación de scope

El SW tiene un scope **implícito** definido por su path. Los SWs registrados sin opción `scope` adoptan el directorio del archivo: `/firebase-messaging-sw.js` → scope `/`.

**Conflicto potencial con `sw.js`:** ambos tendrían scope `/`. Esto **NO es un problema** porque:
1. El SDK de Firebase registra el SW con scope explícito `/firebase-cloud-messaging-push-scope`, no con scope raíz.
2. Cada SW maneja eventos distintos: `sw.js` → `fetch`, `notificationclick`. `firebase-messaging-sw.js` → `push`, `notificationclick`.
3. El navegador permite múltiples SWs en distintos scopes simultáneamente.

**Verificación:** en Chrome DevTools → Application → Service Workers, debes ver **dos** SWs activos:
- `sw.js` con scope `https://tu-dominio/`
- `firebase-messaging-sw.js` con scope `https://tu-dominio/firebase-cloud-messaging-push-scope`

## 5.8 — Versionado del SW para invalidación

Cuando cambies la config Firebase o la lógica del `notificationclick`, los navegadores ya tienen cacheado el SW antiguo. Para forzar update:

**Opción adoptada:** que el script generador inserte una constante `SW_VERSION` con timestamp del build:

```js
// Dentro del SW generado
const SW_VERSION = '${new Date().toISOString()}';
```

El navegador detecta cambio en el contenido del SW (aunque sea un comentario) y descarga la versión nueva. La activación inmediata se garantiza con `skipWaiting + clients.claim`:

```js
self.addEventListener('install', () => self.skipWaiting());
self.addEventListener('activate', (e) => e.waitUntil(self.clients.claim()));
```

(Estas líneas se incluyen en el template del script generador — agregarlas si no están.)
